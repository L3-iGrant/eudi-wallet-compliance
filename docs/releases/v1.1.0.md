# v1.1.0: ISO/IEC mdoc/mDL profile

Released: TBD (Phase 7 of the EUDI Wallet Compliance Self-Assessment build).

## What changed

The Self-Assessment runner now accepts the second EAA profile defined by ETSI TS 119 472-1 v1.2.1: ISO/IEC mdoc, covering both mobile driving licenses (mDL, docType `org.iso.18013.5.1.mDL`) and non-mDL mdocs that follow ISO/IEC 23220-2.

Concretely, this release ships:

- A CBOR + COSE_Sign1 parser for mdoc payloads (`packages/shared/src/mdoc.ts`), built on `cbor-x`. Decodes the `IssuerSigned` envelope, the four-tuple COSE_Sign1, and the tag(24)-wrapped `MobileSecurityObject` into a typed `ParsedMdoc` shape.
- 105 new conformance checks under `packages/engine/src/checks/eaa-6-*.ts`, mirroring the SD-JWT VC §5 layout: §6.1 general requirements (8), §6.2.2–§6.2.4 metadata and identifiers (36), §6.2.5–§6.2.9 subject, issuance, validity and constraints (32), §6.2.10–§6.6 status, attested attributes, key binding and signature posture (29).
- A runtime status resolver (`EAA-6.2.10.2-01`) that mirrors the SD-JWT VC equivalent. Reads the status URI and index from the MSO with the IETF nested-envelope tolerance carried over from v0.1, fetches the Token Status List, and reads back the registered status value.
- An mdoc-aware upload form: the EAA payload field accepts hex or base64-encoded CBOR; the file dropper accepts `.cbor`, `.cbor.base64` and `.b64`. Auto-fill pulls the issuer cert from `x5chain` and the status URL from `MSO.status`; the type-metadata textarea is replaced with read-only docType and namespace badges. A soft warning surfaces when the auto-detected payload format disagrees with the chosen scope.
- 5 new mdoc reference samples (`packages/controls/data/reference-samples/mdl-eaa-{1..5}.json`), each carrying genuinely encoded CBOR (placeholder signatures), exercising distinct features: mDL baseline, mDL with TokenStatusList, non-mDL with category, mdoc QEAA with the qualified URN, mdoc PuB-EAA with `shortLived=true`.

Behind the scenes, two refactors landed first as a behaviour-preserving substrate:

- Payload parsing was lifted from inside each check up to `runAssessment`. Checks now receive a tagged-union `ParsedEvidence` and dispatch on `kind === 'sd-jwt-vc' | 'mdoc'`. No SD-JWT VC verdict shape changed.
- `runAssessment` now filters the registry by `scope.profile ∩ control.profile` so an mdoc payload no longer triggers spurious `na` rows from SD-JWT VC checks.

## What to try

1. Visit `/eudi-wallet-compliance/modules/eaa-conformance/profiles/mdoc/` to see the new mdoc-tagged controls in the catalogue (100 catalogue entries plus 5 synthetic ones for the resolver and member-presence rules).
2. Visit `/eudi-wallet-compliance/modules/eaa-conformance/reference-samples/` and click any of the `MDL-EAA-*` samples. The detail page renders the CBOR (base64) plus the decoded MSO and namespaces.
3. Click "Run Self-Assessment with this sample" on an mdoc sample to land on the upload form pre-filled. Hit Run Assessment.
4. Or paste your own mdoc: drop a `.cbor` file into the upload page, or paste the base64 form into the textarea. The auto-fill helper extracts the issuer chain and status URL, the docType + namespace list show as badges.

## Known limitations

- **Cryptographic signature verification is deferred** for both SD-JWT VC and mdoc. The COSE_Sign1 wrapper structure is parsed and the protected-header members (`x5u`, `x5t`, `x5chain`) are checked; verifying the signature itself awaits trust-list integration.
- **`x5t` algorithm field is not surfaced.** The parser reads only the hash bytes from the RFC 9360 `x5t = [alg, hash]` shape. The QEAA / PuB-EAA "x5t digest algorithm shall be SHA-256" rules (`-6.6.2-03` / `-6.6.3-03`) therefore return `na` with a parser-extension note.
- **mdoc reference sample signatures are placeholders.** The CBOR is genuinely encoded (the `cbor_base64` field round-trips through `parseMdoc` and exposes the structures the descriptions claim), but the COSE signature is 64 zero bytes. This matches the engine's "structural-only" verification posture; downstream verifiers that expect valid signatures will fail.
- **Engine cannot recompute valueDigests.** Disclosure-to-digest binding (per ISO 18013-5 §9.1.2.5) is not yet verified. Each disclosed `IssuerSignedItem` is taken at face value.

## Decisions surfaced for review

- The four `EAA-6.2.10.1-{04,06,08,10}` member-presence rules and `EAA-6.2.10.2-01` runtime resolver are synthetic catalogue entries (not literal §6 spec text). They mirror the SD-JWT VC §5.2.10.1 numbering for ergonomic parity. If ETSI publishes a v1.3 with explicit per-member presence rules, the catalogue should re-anchor these IDs to the published clause.
- Clause-4 cross-cutting controls keep `profile: [abstract]` rather than `[sd-jwt-vc, mdoc]`. Functionally equivalent under the current dispatch (the scope filter treats `abstract` as cross-cutting); the abstract tag future-proofs for clauses 7 (W3C VC) and 8 (X.509-AC).
- The upload form reuses the existing `Evidence.eaaPayload: string` slot rather than introducing a binary channel. Binary CBOR is base64-encoded on read so the existing string-typed plumbing carries through to the engine without ripple changes.

## What's next

- Trust-list integration for the deferred signature-verification rules (both SD-JWT VC and mdoc).
- A truly distinct mdoc sample generator with valid COSE signatures, replacing the current placeholder-signature substrate.
- Disclosure-to-digest binding verification in the engine.
- Optional support for clauses 7 (W3C VC) and 8 (X.509-AC) of TS 119 472-1, following the same modular pattern.
